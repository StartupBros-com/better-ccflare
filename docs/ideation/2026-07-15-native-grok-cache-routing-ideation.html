Evidence before direction

Grounding Context

Current xAI path

XaiProvider is a thin generic adapter: Messages becomes Chat Completions, every Claude family defaults to grok-4.3, and the only post-conversion specialization requests streaming usage. It emits no native Grok affinity identifier.

Codex precedent

The fork already derives conversation-partitioned prompt_cache_key values, gates them by real endpoint, traces cache outcomes, runs pacing canaries, and correctly separates cached from uncached input usage.

Routing mismatch

Session affinity can pin a Claude client session to an account and restore it after temporary failover, but force-route can silently fall back. Account locality and provider-native cache identity are not one contract.

xAI native behavior

xAI documents automatic exact-prefix caching. Chat Completions uses x-grok-conv-id; Responses uses prompt_cache_key. Chat continuation requires unchanged reasoning_content, and cached tokens have protocol-specific usage paths.

Prefix risk

The shared conversion path does not provide a versioned Grok prefix contract. Tool ordering, schema representation, compaction, session resume, reasoning replay, model fallback, and proxy mutations can each change a cacheable prefix.

Forensic leverage

Claude Code 2.1.207 and older local binaries are available for differential inspection, and the repo already contains compaction compatibility work. This can become a sanitized fixture corpus without scripted traffic to real Anthropic accounts.

Central finding: the immediate gap is not speculative cache tuning. The official xAI Chat affinity mechanism is documented, yet better-ccflare does not emit it or verify its cached-token result. The highest-value first move is therefore a narrow end-to-end proof, instrumented from day one.

1Grok Cache-Native Vertical Slice

Description: Ship the smallest coherent native cache path on official xAI Chat: derive a privacy-safe conversation identity using the Codex conversation-partitioning lesson, attach it as x-grok-conv-id, bind it to the selected xAI account, and prove streaming and non-streaming cached_tokens translation with fixtures. Record native ID hash, serving account, prefix fingerprint, and cache outcome from the first canary. Keep custom endpoints gated off until capability is verified.

One end-to-end path proves identity, locality, native transport, and measurement together.

Axis
Native affinity and account locality
Basis
direct: packages/providers/src/providers/xai/provider.ts has no Grok cache identity, while packages/providers/src/providers/codex/provider.ts already derives stable per-conversation keys and session-affinity.ts already models account locality. external: xAI documents x-grok-conv-id for Chat.
Rationale
This is the fastest route from documented provider capability to falsifiable production value. It avoids building generalized scheduling or protocol infrastructure before native mechanics are proven.
Downsides
Conversation partitioning must be specified carefully, official endpoint gating is required, and a useful hit-rate experiment needs enough repeated-prefix traffic. It does not yet solve compaction, Responses, or complex failover.

2Grok Cache Flight Recorder and Experiment Plane

Description: Generalize the narrow, proven parts of the Codex cache trace into a provider-neutral event model, with Grok as the second implementation. Record conversation hash, native affinity hash, serving account, protocol, model, prefix-component fingerprints, cache epoch, cached and uncached tokens, first-token latency, failover state, and whether telemetry is absent rather than zero. Support deterministic conversation-level cohorts for affinity on/off, prefix variants, and later Chat versus Responses comparisons.

The recorder turns one cache percentage into causal dimensions an operator can act on.

Axis
Cache observability and experimentation
Basis
direct: Codex already has JSONL traces, HMAC fingerprints, pacing canaries, and cohort analysis; shared request storage already separates cache-read and uncached input. xAI currently relies on the generic cached_tokens path, with no xAI-specific fixture or diagnosis.
Rationale
Every later optimization depends on distinguishing identity drift, account drift, prefix mutation, protocol change, and unavailable telemetry. This idea compounds by making both successful and failed experiments durable knowledge.
Downsides
Trace schemas can grow into a second analytics system. Privacy-safe fingerprinting, sampling, storage bounds, and statistical minimums need discipline. The first iteration should stay Grok-focused rather than redesign every provider.

3Claude-to-Grok Cache Compatibility Lab

Description: Turn binary analysis into a bounded compatibility system, not an open-ended reverse-engineering project. Differentially inspect installed Claude Code versions and generate sanitized fixtures for normal turns, tools, sibling agents, subagents, resume, and compaction. Replay them through a versioned Grok Prefix ABI at the incoming Claude request, normalized representation, and final Grok wire stages; add a protocol capability matrix for official xAI Chat, official Responses, and unverified custom endpoints.

Binary evidence becomes reusable fixtures, then a release gate for cache-relevant drift.

Axis
Exact-prefix fidelity and Claude Code request stability
Basis
direct: local Claude Code 2.1.207 and older binaries are available; compaction compatibility work already exists; the xAI test suite omits tools, reasoning, images, cache controls, unsupported parameters, and exact-prefix stability. external: xAI exact-prefix behavior and community cache-fix work make structural stability load-bearing.
Rationale
The lab converts one-time forensics into a compounding release corpus. It also prevents unsafe canonicalization by requiring every normalization to prove semantic equivalence against observed Claude behavior.
Downsides
Capturing behavior without real Anthropic calls requires careful local stubbing and fixture design. Binary formats can change, and the lab must not become an obligation to support every historical Claude release.

4Cache Lineage State Machine

Description: Replace invisible best-effort cache continuity with explicit lineage states: warm owner, temporarily displaced, restored, cold migrated, and indeterminate after a partial stream. Offer bounded policies such as strict, restore, and migrate. Every transition states which account owns the lineage, whether native identity can be retained, whether the request is safe to retry, whether reasoning and tools are acknowledged, and what the operator sees.

Availability becomes an explicit continuity decision instead of a silent cache break.

Axis
Failover, continuation, and operational cache semantics
Basis
direct: account and model retries can move a request; session affinity preserves the original mapping after temporary failover; force-route can fall back; a started stream cannot transparently move. external: xAI Chat continuation depends on exact reasoning history.
Rationale
The proxy currently optimizes the next request while hiding the state cost to the rest of the agentic conversation. Explicit lineage semantics let availability and cache continuity become intentional policy choices.
Downsides
This changes observable routing behavior and may trade availability for continuity in strict mode. It cannot guarantee prevention of external tool duplication after an ambiguous partial response, so acknowledged boundaries must remain conservative.

5Native Grok Responses Route

Description: Add a first-class official-xAI Responses path using the existing Responses adapter foundation, with prompt_cache_key, native usage mapping, reasoning and tool conformance fixtures, and protocol pinning for the life of a cache epoch. Keep Chat as the established baseline and as the compatibility path for custom endpoints. Compare matched cohorts only after the Chat vertical slice and recorder are trustworthy.

Axis
Grok protocol adaptation
Basis
direct: xAI currently routes only through Chat Completions, while the monorepo already has a Responses adapter and Codex demonstrates a production Responses conversion. external: xAI exposes prompt_cache_key for Responses and a different native identifier for Chat.
Rationale
Responses may offer cleaner native continuation and cache control than indefinite Chat emulation. Keeping both paths creates a measured protocol portfolio rather than a risky one-way migration.
Downsides
High conformance burden across tools, reasoning, images, stream events, usage, and custom endpoints. Protocol switching mid-conversation must be forbidden or treated as a cold epoch. This should not precede a measured Chat baseline.

6Compaction-Aware Cache Epochs

Description: Detect reliably supported Claude Code compaction and resume boundaries, mark them as cache epochs, and classify the resulting prefix rewrite separately from ordinary misses. Rotate native identity only when evidence shows an epoch represents a different cache lineage, not simply because a compaction marker exists. Use the compatibility lab to compare versions and identify which stable prefix portions, if any, survive.

Axis
Exact-prefix fidelity and lifecycle
Basis
direct: the repository has Claude Code 2.1.207 compaction compatibility evidence and preserves the pre-transform request body; multiple local CLI versions permit differential study. external: compaction and resume are recurring exact-prefix breakpoints in Claude Code cache investigations.
Rationale
Compaction happens when prompts are large and misses are expensive. Explicit epochs make legitimate rewrites visible and prevent them from contaminating measurements of proxy-induced drift.
Downsides
The current evidence does not yet prove a reliable cross-version boundary signal. Premature key rotation could discard useful cache locality, so this remains research-led and ranks below the core path.
